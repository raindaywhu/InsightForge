# Prompt 设计决策文档

> 记录 InsightForge Agent 和 Task 的 Prompt 设计思路与决策

---

## 1. 设计原则

### 1.1 核心原则

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **角色明确** | Agent 有清晰的身份定位 | 明确的 role + backstory |
| **目标导向** | 每个任务有具体的目标 | goal 定义期望产出 |
| **结构化** | 复杂任务分解为步骤 | description 中分步骤指导 |
| **可验证** | 输出格式明确 | expected_output 定义格式 |

### 1.2 设计方法论

```
Agent 设计 = 角色定位 + 能力边界 + 行为约束
Task 设计 = 任务描述 + 执行步骤 + 输出规范
```

## 2. Agent 设计决策

### 2.1 Technical Analyst（技术分析师）

#### 设计思路

**定位**：深度研究者，而非简单的内容生成器

**为什么选择"分析师"而非"研究员"？**
- 分析师强调输出有价值洞察，而非单纯的信息汇总
- 更符合产品定位：提供可操作的建议

#### Role 设计

```yaml
role: 技术分析师
```

**决策**：简洁直接，避免过度修饰

#### Goal 设计

```yaml
goal: 深度研究分析主题，运用学术分析方法论，输出高质量、有洞察力的技术报告
```

**关键决策**：
- 强调"深度"而非"快速"
- 明确方法论来源（学术分析）
- 定义输出质量标准（高质量、有洞察力）

#### Backstory 设计

```yaml
backstory: |
  你是一位资深的技术分析师，拥有丰富的学术研究经验和行业洞察力。
  
  你擅长运用各种分析框架和方法论：
  - SWOT 分析：评估优势、劣势、机会、威胁
  - PEST 分析：分析政治、经济、社会、技术环境
  - 波特五力：分析行业竞争格局
  - 价值链分析：梳理价值创造过程
  
  你的分析总是：
  - 数据支撑充分，引用可靠来源
  - 逻辑清晰，论证有力
  - 洞察独到，不流于表面
  - 结构严谨，易于理解
  
  你撰写的报告遵循技术报告标准格式，包含执行摘要、主体分析、结论建议等部分。
```

**设计要点**：

| 要点 | 目的 |
|------|------|
| 明确分析方法论 | 引导 Agent 使用专业框架 |
| 列举具体框架名 | 与知识库内容匹配，便于 RAG 检索 |
| 定义分析质量标准 | 约束输出质量 |
| 规定报告格式 | 确保输出结构化 |

#### 配置参数

```yaml
verbose: true        # 开启详细日志，便于调试
allow_delegation: false  # 禁止任务转派，保持责任清晰
memory: true         # 启用记忆（暂禁用，待解决兼容性）
```

### 2.2 Presentation Designer（演示设计师）

#### 设计思路

**定位**：信息架构师，而非美工

**为什么需要独立的 PPT Agent？**
- 技术报告和 PPT 是两种不同的信息载体
- 报告重深度，PPT 重表达
- 分离职责，各司其职

#### Role 设计

```yaml
role: 演示设计师
```

#### Goal 设计

```yaml
goal: 将技术报告转化为专业、美观、有说服力的PPT演示文稿
```

**关键决策**：
- 明确输入（技术报告）和输出（PPT）
- 定义三个质量维度：专业、美观、有说服力

#### Backstory 设计

```yaml
backstory: |
  你是一位资深的演示设计专家，精通视觉传达和信息设计。
  
  你深谙PPT设计核心原则：
  - 简洁：每页一个核心观点，避免信息过载
  - 聚焦：突出关键信息，引导观众注意力
  - 层次：清晰的视觉层次，便于信息吸收
  - 一致：统一的风格和配色，专业美观
  
  你擅长：
  - 信息架构：提炼报告核心，设计清晰的信息流
  - 视觉呈现：选择合适的图表类型，增强说服力
  - 配色排版：运用设计原则，创造专业美感
  
  你的PPT设计总是让复杂的报告内容变得清晰易懂...
```

**设计要点**：

| 要点 | 目的 |
|------|------|
| 定义设计原则 | 约束 PPT 风格 |
| 列举核心技能 | 引导 Agent 关注关键能力 |
| 强调输出效果 | "复杂变清晰" |

## 3. Task 设计决策

### 3.1 Analyze Task（分析任务）

#### Description 结构

```
1. 明确研究范围
2. 检索分析框架
3. 执行深度分析
4. 撰写技术报告
```

**设计决策**：

| 步骤 | 目的 | 与知识库关联 |
|------|------|-------------|
| 明确研究范围 | 避免跑题 | 无 |
| 检索分析框架 | 使用专业知识 | SWOT/PEST/波特五力 |
| 执行深度分析 | 核心推理 | 分析方法论 |
| 撰写技术报告 | 结构化输出 | 报告模板 |

#### Expected Output 设计

```yaml
expected_output: |
  一份完整的Markdown格式技术报告，包含：
  - 标题和执行摘要
  - 完整的分析过程
  - 清晰的结论和建议
  - 适当的图表说明（使用Markdown图表语法）
```

**关键决策**：
- 指定 Markdown 格式 → 便于后续处理
- 列出必要组件 → 确保结构完整
- 允许图表 → 增强表达力

### 3.2 Design Task（设计任务）

#### Context 依赖

```yaml
context: [analyze_task]
```

**关键决策**：design_task 依赖 analyze_task 的输出

#### 输出格式约束

```yaml
expected_output: |
  一份详细的PPT结构描述，JSON格式，包含完整的slides数组：
  {
    "metadata": {...},
    "slides": [...]
  }
```

**关键决策**：
- 强制 JSON 格式 → 便于程序解析
- 定义结构 schema → 确保字段完整
- 这对后续 PPTGenerator 至关重要

## 4. 已知问题与优化方向

### 4.1 当前限制

| 问题 | 影响 | 状态 |
|------|------|------|
| Memory 兼容性 | 尝试连接 OpenAI API | 暂禁用 |
| 知识库 RAG | 已解决（DashScope Embedding） | ✅ |
| LLM 输出 JSON 格式 | 偶尔有引号转义问题 | 已修复 |

### 4.2 未来优化方向

#### Prompt 优化

1. **Few-shot 示例**：在 Task description 中添加示例输出
2. **思维链引导**：增加 CoT 提示
3. **质量检查点**：添加自检环节

#### Agent 能力增强

1. **工具集成**：允许 Agent 调用搜索 API
2. **多轮对话**：支持用户反馈迭代
3. **并行分析**：多 Agent 协作
